PASS 4 PROMPT — Intervention Mapping

You are given a software failure mechanism extracted from a support ticket.

Your task is to determine the corrective action that would eliminate or prevent this failure.


INPUT

mechanism:
{{mechanism}}


DEFINITIONS

Mechanism
The internal system behavior that causes the failure.

Intervention
A specific system change that removes or prevents the failure mechanism.


MECHANISM CLASS TAXONOMY

Select exactly one:

- calculation_logic_error
- schema_mismatch
- data_validation_failure
- configuration_mismatch
- authentication_failure
- integration_mapping_error
- integration_communication_failure
- state_inconsistency
- synchronization_failure
- dependency_missing
- field_mapping_error
- cache_inconsistency
- permission_error
- other                       (use ONLY when none of the above fit; requires proposed_class)


MECHANISM INTERPRETATION RULES

Classify based on the SYSTEM FAILURE PATTERN, not surface wording.

Guidance:

- calculation_logic_error → incorrect calculations, selection logic, or formula behavior
- data_validation_failure → invalid input not rejected or validated
- schema_mismatch → structural mismatch between expected and actual data shape
- field_mapping_error → incorrect mapping between source and target fields (typically import/export/integration)
- state_inconsistency → system state is missing, stale, not persisted, not propagated, or not correctly associated across components
- synchronization_failure → data not updated across systems or processes in time
- integration_mapping_error → incorrect mapping between internal system and external system fields
- integration_communication_failure → API, network, transport, or connection failures
- configuration_mismatch → system configured incorrectly
- dependency_missing → required data, reference, or resource is not present

If the issue involves missing context, lost references, incorrect association between entities, or failure to carry required identifiers/details through the workflow, prefer:
→ state_inconsistency

Use field_mapping_error only when the failure is specifically about incorrect source-to-target field mapping.


INTERVENTION TYPES

Select exactly one:

- software_fix            (code change)
- configuration_change    (system configuration correction)
- validation_guardrail    (prevent invalid input)
- integration_fix         (external system/interface fix)
- data_repair             (fix corrupted data)
- documentation           (missing documentation)
- customer_training       (user workflow correction)
- other                   (use ONLY when none of the above fit; requires proposed_type)


INTERVENTION GUIDELINES

- The intervention must directly address the mechanism.
- The intervention must describe ONE specific system change.
- Prefer structural fixes over temporary workarounds.
- Do NOT describe troubleshooting steps.
- Do NOT mention tickets, users, or support processes.

INTERVENTION NORMALIZATION RULES

The intervention_action must be reusable across similar failures.

Rules:

- Remove excessive specificity
- Do NOT include customer-specific, ticket-specific, or one-off contextual details
- Prefer generalized corrective changes over narrow case-specific fixes
- Focus on the underlying system change, not the exact incident wording
- Keep the action concise and structurally reusable across similar tickets
- intervention_action MUST be 12 words or fewer

Normalization guidance:

- Remove unnecessary qualifiers (e.g., "reading verification process", "specific report")
- Prefer core system action + target

Examples:

"update reading verification process to correctly persist account association with error details"
→ "persist account association in error records"

"update billing logic to filter inactive customer records"
→ "exclude inactive records in selection logic"

"add validation to reject negative readings during import"
→ "validate and reject invalid input values"

Prefer shorter, reusable actions that can cluster with semantically similar interventions.

ACTION VERB REQUIREMENTS

Use precise, deterministic verbs such as:

- add
- update
- enforce
- validate
- reject
- map
- persist
- retry
- correct
- align
- normalize

Avoid vague verbs such as:

- resolve
- ensure
- handle
- fix
- address
- improve


STRUCTURE

Write the intervention as:

<component or subsystem> + <specific change>

Examples:

- add schema validation to import parser to reject invalid file formats
- update billing calculation logic to prevent duplicate service charge application
- enforce API credential validation in integration authentication service
- correct field mapping logic between payment method and account description


OUTPUT FORMAT

Return JSON only.

{
  "mechanism_class": "",
  "intervention_type": "",
  "intervention_action": ""
}

If mechanism_class is "other", also include:
  "proposed_class": "<snake_case name for the new class>"

If intervention_type is "other", also include:
  "proposed_type": "<snake_case name for the new type>"


TASK

Determine the correct mechanism_class, intervention_type, and intervention_action for the given mechanism.
