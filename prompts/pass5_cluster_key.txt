You are normalizing technical root causes into reusable clustering keys.

Task:
Convert the mechanism into a short, standardized label.

The cluster_key MUST represent the underlying failure pattern and MUST be reusable across many tickets.


--------------------------------
CORE OBJECTIVE
--------------------------------

Maximize grouping across similar failures.

Prefer FEWER, broader clusters over many specific ones.

If two mechanisms differ only by object detail, they MUST map to the same cluster_key.


--------------------------------
HARD CONSTRAINTS
--------------------------------

- Output MUST be a single snake_case phrase
- Max 6 words
- NO punctuation except underscores
- NO product names
- NO implementation-specific details
- NO variable names, field names, or identifiers unless essential
- NO verbs like: update, fix, handle, process
- MUST describe a failure pattern, not a specific instance


--------------------------------
ANTI-FRAGMENTATION RULE (CRITICAL)
--------------------------------

If the mechanism contains a specific object (account, employee, meter, payment, etc.):

→ REMOVE the object unless it changes the failure category

Examples:

- missing_employee_hours_calculation
- missing_payment_amount_calculation
→ BOTH become:
missing_calculation_logic

---

- missing_account_identifier_in_error
- missing_customer_id_in_error
→ BOTH become:
error_not_linked_to_entity

---

- meter_not_linked_to_account
- payment_not_linked_to_customer
→ BOTH become:
entity_not_linked

Use generic placeholders when possible:
- entity
- record
- data
- value


--------------------------------
CANONICAL PATTERNS (PREFERRED)
--------------------------------

Always map to one of these if possible:

- entity_not_linked
- wrong_entity_selected
- missing_entity_association
- duplicate_records_created
- invalid_state
- state_not_persisted
- state_not_propagated
- missing_validation
- missing_idempotency_validation
- incorrect_calculation_logic
- incorrect_configuration_used
- invalid_configuration_state
- missing_dependency
- invalid_input_handling
- stale_state_used
- inconsistent_state


--------------------------------
NORMALIZATION RULES
--------------------------------

Map aggressively:

- "fails to associate X with Y"
  → entity_not_linked

- "missing identifier / reference"
  → entity_not_linked

- "wrong record / account / meter selected"
  → wrong_entity_selected

- "duplicate creation / applied twice"
  → duplicate_records_created

- "missing validation"
  → missing_validation

- "missing idempotency"
  → missing_idempotency_validation

- "incorrect logic / miscalculation"
  → incorrect_calculation_logic

- "configuration issue"
  → invalid_configuration_state

- "not saved / not retained"
  → state_not_persisted

- "not reflected / not updated downstream"
  → state_not_propagated

- "stale / cached / outdated data"
  → stale_state_used


--------------------------------
GENERALIZATION PRIORITY
--------------------------------

When choosing between:

- Specific → generic
- Narrow → broad
- Descriptive → reusable

ALWAYS choose:

→ generic, broad, reusable

INPUT

{{mechanism}}

OUTPUT

Return ONLY the cluster key:

<snake_case_key_only>
